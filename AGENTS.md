# 项目级 Agent 规则

## 交付与提交

- 每次完成一个明确的需求任务后，如已完成与改动直接相关的验证，应自动执行 `git commit`
- 若关键验证无法执行，提交说明中应如实反映限制，不得伪称已通过
- 提交信息格式：`<type>(scope): <summary>`
- `summary` 使用中文、动词开头、长度不超过 50 字、不加句号
- 提交后使用 `git push origin` 主动推送

## 事务边界

- 所有业务事务只允许在 `store` 层开启
- `service`、`handler`、`app` 等上层不得直接开启数据库事务
- 若一个用例需要原子性，应优先新增或扩展 `store` 方法，由 `store` 统一封装事务边界

## 近期经验沉淀

### 经验：事务内业务执行按“事务骨架 + 纯计算”拆分

- **触发信号**：`store` 文件快速膨胀，事务方法同时包含查询、校验、金额计算、状态推进与持久化
- **根因 / 约束**：事务原子性要求业务执行留在 `store`，但纯计算继续堆在 `store` 会让文件过重、测试粒度变粗、职责边界变模糊
- **正确做法**：`store` 只保留“读 → 调纯函数 → 写”的事务骨架；将无副作用的校验、金额比较、状态推进、说明文案生成等逻辑下沉到独立纯计算模块或纯函数中（如 `internal/bookkeeping` 等），由 `store` 在事务内调用；目录名仅作示例，不作强制要求
- **验证方式**：纯计算逻辑单测放在对应纯模块；事务成功/失败路径继续保留 `store` 集成测试
- **适用范围**：计划任务执行、批量状态推进、余额结算、分期/周期记账等需要事务但又包含较多纯规则计算的场景

### 经验：共享归一化逻辑挂运行时或包级函数，不创建空壳 service 复用方法

- **触发信号**：代码里出现 `(&SomeService{...}).method(...)` 这类临时构造，仅为了复用一个校验/归一化函数
- **根因 / 约束**：该方法实际依赖的是共享运行时或纯参数，而不是具体 service 语义；把它留在 service 上会诱导错误归属并制造空壳实例
- **正确做法**：若逻辑依赖共享 store / settings / policy，则挂到 `runtime`；若只依赖入参，则提为包级纯函数；禁止通过构造零值或临时 service 实例复用方法
- **验证方式**：调用方改为直接依赖 `runtime` / 包级函数；补针对归一化函数的定向测试，覆盖默认值、边界值与错误路径
- **适用范围**：输入归一化、权限前置校验、金额标准化、导出参数解析、计划任务 payload 构造

### 经验：`store` 的单条查询未找到统一归一化为 `store.ErrNotFound`

- **触发信号**：上层代码同时判断 `gorm.ErrRecordNotFound` 与 `store.ErrNotFound`，或者不同 store 的 `GetByID` / `GetBy...` 行为不一致
- **根因 / 约束**：底层 ORM 错误直接外泄会把持久化实现细节泄漏到 `service` / `handler`，迫使上层写重复分支
- **正确做法**：所有返回单条记录的 `store` 查询（如 `GetByID`、`GetByClientID`、`GetByUserAndClient`、`GetMember` 等）统一经 `normalizeNotFound` 返回 `store.ErrNotFound`
- **验证方式**：在 `store/not_found_test.go` 集中覆盖；上层只依赖 `errors.Is(err, store.ErrNotFound)`，不再直接判断 ORM 未找到错误
- **适用范围**：所有单记录读取接口；列表查询、聚合查询、存在性探测若有特例需在注释中明确说明

## 工程质量硬门禁

- **KISS**：默认选择能解决当前问题的最简单方案，避免为了“未来可能需要”引入框架、抽象层、配置项或通用机制
- **YAGNI**：没有明确需求、调用方、验收标准或近期使用场景的能力不要实现；不得预留未使用的接口、字段、开关或扩展点
- **NO OVER-ENGINEERING**：抽象必须服务真实重复、真实边界或真实复杂度；新增抽象前先确认现有直接实现确实已经难以维护
- **NO PARTIAL IMPLEMENTATION**：不得交付半成品、空实现、假实现、只接 UI 不接后端、只写 happy path 或留下未说明的功能缺口
- **NO CODE DUPLICATION**：新增函数、常量、类型或组件前必须先检索并阅读现有实现，优先复用；确需重复时应说明原因并尽量收敛为共享能力
- **NO DEAD CODE**：未使用的函数、类型、字段、路由、组件、配置、测试辅助和文档入口必须删除；不要保留“以后可能用”的代码
- **IMPLEMENT TEST FOR EVERY FUNCTIONS**：新增或修改有业务行为的函数必须补充对应测试；纯装配、薄包装或无法直接测试的函数至少通过调用方、集成测试或路由测试覆盖
- **NO CHEATER TESTS**：测试必须模拟真实调用路径，覆盖关键路径、边界与错误场景；不得只断言 mock 被调用、只测实现细节、写不会失败的空测试或为通过测试而降低断言质量
- **NO MIXED CONCERNS**：保持分层职责清晰；handler 只处理 HTTP 解析与响应，service 承载业务编排，store 封装持久化和事务，UI 组件不得混入数据持久化或后端规则
- **NO RESOURCE LEAKS**：创建资源必须有对应释放路径；数据库连接、事务、响应体、文件句柄、定时器、goroutine、事件监听器、订阅和临时文件都必须正确关闭或清理
- **Verify Facts**：事实性结论必须来自代码、测试、日志、文档或可靠外部来源；不确定时明确标注假设，不得把猜测、记忆或推断包装成事实

## 后端函数注释

- 后端注释以高信噪比为准，解释“为什么、业务契约、边界条件、副作用、错误语义、事务/权限/安全约束”，不要复述代码已经表达清楚的内容
- 对导出符号使用 Go 风格注释，但不强制套用“入参 xxx 为...；返回值为...”模板；只有当参数或返回值含义不直观、存在单位/枚举/权限/幂等/错误约定时才展开说明
- 对未导出函数，仅在承载业务规则、复杂算法、跨层约束、非显然副作用或容易误用时补充注释；简单 getter、setter、构造器、薄包装、测试辅助和显而易见的私有函数可不写注释
- 注释应优先说明调用方需要知道的契约，而不是实现步骤；若代码结构能让意图自解释，应优先重命名、拆函数或收敛逻辑，而不是用长注释解释坏代码
- 注释要求简洁准确，避免空泛描述、重复函数名、逐字段机械解释和“为了满足规则”的低价值注释
- 注释使用中文
- 不对存量代码做纯注释式批量补齐；仅在函数被本次需求修改且确有说明价值时顺手补注释
- 所有本次新增的后端导出符号和有业务含义的新增函数，提交前必须补齐必要中文注释；无业务语义的薄包装函数可通过清晰命名替代注释
