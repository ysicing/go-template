# 用户管理 CRUD 子功能设计说明

## 1. 概述

本设计用于在现有 Go 全栈模板基础上补齐“用户管理 CRUD 示例”子功能，目标是提供一个可复用、边界清晰、适合作为模板示例的后台用户管理子系统。

本次子功能范围包括：

- 管理员用户列表
- 用户详情查看
- 新建用户
- 编辑用户
- 启用 / 停用用户
- 软删除用户
- 当前登录用户修改自己的密码

本次明确不包含：

- 邮件找回密码完整流程
- 完整 RBAC
- 恢复已软删除用户
- 审计日志
- 密码历史记录

## 2. 总体前端架构约束

### 2.1 总体布局

前端整体布局框架保持一致，不为不同子系统重复设计完全独立的后台壳。统一后台布局至少包含：

- 顶部栏
- 左侧导航区域
- 页面标题 / 面包屑区域
- 主内容区域
- 主题、语言、用户菜单

### 2.2 子系统设计

前端按“子系统”组织业务代码，而不是仅按单页文件堆叠：

```text
web/src/
├── app/
│   ├── router/
│   ├── providers/
│   └── layouts/
│       ├── public-layout.tsx
│       ├── setup-layout.tsx
│       └── admin-layout.tsx
├── shared/
│   ├── ui/
│   ├── lib/
│   ├── hooks/
│   └── navigation/
└── subsystems/
    ├── auth/
    ├── admin-users/
    └── system-settings/
```

其中：

- 全局布局保持统一
- 各子系统提供自己的导航配置与业务页面
- 左侧导航可根据当前子系统切换内容，但不改变整体后台布局骨架

### 2.3 通用组件策略

通用组件优先使用 `shadcn components`，统一放在 `shared/ui/`。子系统只在此基础上做业务组合，不自建平行的通用组件体系。

适合作为通用层的组件包括：

- `Button`
- `Input`
- `Dialog`
- `Sheet`
- `Table`
- `Badge`
- `Form`
- `Select`
- `DropdownMenu`
- `Tabs`
- `Card`

业务组件放在各子系统的 `components/` 中，例如：

- 用户筛选栏
- 用户表格
- 用户编辑表单
- 修改密码表单

## 3. 数据模型

本次复用现有 `users` 表，不新增复杂表结构。

### 3.1 复用字段

- `username`
- `email`
- `password_hash`
- `role`
- `status`
- `last_login_at`
- `deleted_at`

### 3.2 字段约束

- `role` 继续只保留：
  - `user`
  - `admin`
- `status` 默认约定：
  - `active`
  - `disabled`
- `deleted_at` 继续使用 GORM 软删除机制

## 4. 后端 API 设计

### 4.1 管理员用户接口

统一挂在 `/api/admin/users` 下：

- `GET /api/admin/users`
  - 查询用户列表
  - 支持参数：
    - `keyword`
    - `role`
    - `status`
    - `page`
    - `page_size`
- `GET /api/admin/users/:id`
  - 查看用户详情
- `POST /api/admin/users`
  - 新建用户
- `PUT /api/admin/users/:id`
  - 更新用户资料
- `POST /api/admin/users/:id/enable`
  - 启用用户
- `POST /api/admin/users/:id/disable`
  - 停用用户
- `DELETE /api/admin/users/:id`
  - 软删除用户

### 4.2 当前用户密码修改接口

- `POST /api/auth/change-password`

该接口仅用于当前已登录用户修改自己的密码。

## 5. 服务层设计

新增 `internal/user/service.go`，作为用户管理的统一服务入口。

### 5.1 服务职责

- 用户列表查询
- 单用户读取
- 创建用户
- 更新用户
- 启用 / 停用
- 软删除
- 当前用户修改密码

### 5.2 输入模型

后端 API 不直接暴露 GORM 模型，统一使用 DTO：

- `ListUsersQuery`
- `CreateUserInput`
- `UpdateUserInput`
- `ChangePasswordInput`

这样后续即使模型字段扩展，也不会把 API 契约一起带乱。

## 6. 业务规则

### 6.1 用户列表

默认支持：

- 关键词搜索：用户名 / 邮箱
- 角色筛选
- 状态筛选
- 分页

本次不纳入“是否已软删除”筛选。

### 6.2 创建用户

创建用户时必须满足：

- 用户名唯一
- 邮箱唯一
- 密码先哈希再落库
- 默认角色与状态由输入明确指定或使用安全默认值

### 6.3 编辑用户

管理员可编辑字段仅包含：

- `username`
- `email`
- `role`
- `status`

本次编辑资料不包含密码修改。

### 6.4 启用 / 停用 / 删除

管理员可执行：

- 启用用户
- 停用用户
- 软删除用户

但存在硬性限制：

- 管理员不能停用自己
- 管理员不能删除自己

此限制必须在服务层实现，不能只依赖前端按钮隐藏。

### 6.5 当前用户修改密码

当前登录用户修改自己的密码时，输入为：

- 旧密码
- 新密码
- 确认新密码

规则如下：

- 必须先校验旧密码
- 新密码与确认密码必须一致
- 更新时只修改当前登录用户自己的 `password_hash`

### 6.6 忘记密码

忘记密码本次只预留扩展位，不实现完整邮件找回流程。

## 7. 前端子系统设计

### 7.1 `admin-users` 子系统

负责后台用户管理示例。

建议目录：

```text
web/src/subsystems/admin-users/
├── api/
├── components/
├── hooks/
├── pages/
├── types/
└── navigation.ts
```

#### 页面能力

- 用户列表页
- 用户详情 / 编辑弹层
- 新建用户弹层

#### 页面元素

- 顶部筛选栏
  - 关键词
  - 角色
  - 状态
- 用户表格
  - 用户名
  - 邮箱
  - 角色
  - 状态
  - 最后登录时间
  - 操作列
- 操作区
  - 新建
  - 查看
  - 编辑
  - 启用 / 停用
  - 删除

#### 交互建议

首版使用：

- 统一后台 layout
- 子系统导航配置
- 右侧抽屉 / 对话框承载详情与表单

不额外拆很多独立子页面，保持模板示例轻量。

### 7.2 `auth` 子系统

负责登录、当前用户状态、改密等与身份相关的功能。

本次新增：

- 当前登录用户修改密码表单

建议放入：

- 个人中心页中的安全设置卡片
  或
- `auth` 子系统的安全设置页

本次优先选择更轻量的方式：放入个人中心页中作为一个独立卡片。

## 8. 导航与布局设计

### 8.1 统一后台布局

统一后台 layout 负责：

- 页面主框架
- 左侧导航承载区
- 顶部操作区
- 主题、语言、用户菜单

### 8.2 子系统导航配置

不同子系统提供自己的导航配置，例如：

- `admin-users`
  - 用户列表
  - 新建用户
- `system-settings`
  - 基础设置
  - 安全设置

这意味着：

- 整体后台体验一致
- 子系统左侧导航内容可不同
- 子系统间仍保持边界清晰

## 10. 风险与取舍

### 10.1 不实现邮件找回

原因：

- 会引入邮件配置、发信服务、找回 token、额外页面与接口
- 超出这次“用户管理 CRUD 示例”的合理范围

### 10.2 不做完整后台框架重构

原因：

- 当前重点是验证“统一后台 layout + 子系统导航”的可行性
- 先为用户管理子系统提供清晰模式，再逐步扩展其他子系统

### 10.3 不恢复软删除用户

原因：

- 当前示例主要验证用户生命周期中的常用管理动作
- 恢复逻辑可在后续需要时补充

## 11. 最终范围总结

本次子功能最终交付目标为：

- 一个可工作的用户管理后台示例子系统
- 一个可工作的当前用户修改密码能力
- 一个符合“统一后台布局 + 子系统导航 + shadcn 通用组件优先”原则的前端结构样板

该子功能重点是为模板提供可复制的后台子系统实现范式，而不是一次性做成完整用户中心平台。
